## In Simple Terms

You build a factory. It has 28 production lines, an operating manual for each one, automated deployment, and three shipping docks. It runs from a single phone.

Now imagine cloning that entire factory — every line, every manual, every automation — to someone else's phone. They fork your 28 repos, get your CLAUDE.md constitutions, and the factory runs identically under their account.

That is **Quantum Jump** — the act of replicating a complete content manufacturing system to a new operator. Not teaching them how to build it. Giving them the running factory.

This has already been executed once: three Google accounts (A/B/C) running the same 28-repository system across different personas. Self-replication proof of concept, completed.
</Part1>

## System Architecture

<Accordion>
  <AccordionItem title="1. The Franchise Package">
  What gets replicated in a Quantum Jump:

  | Component | Count | Portability |
  |-----------|-------|-------------|
  | GitHub repositories | 28 | Fork + rename |
  | CLAUDE.md constitutions | 23 | Copy (universal rules) |
  | GitHub Actions CI | per repo | Account-agnostic |
  | Sync scripts | 4+ patterns | Path variables only |
  | Deployment pipelines | 3 stations | Domain swap |
  | Character IP | per persona | Transfer or create new |
  </AccordionItem>

  <AccordionItem title="2. What Makes It Portable">
  The system is portable because of deliberate design choices:

  - **No vendor lock-in**: GitHub + Termux + Claude Code. All replaceable.
  - **Constitution over configuration**: CLAUDE.md defines behavior, not config files.
  - **Script pattern replication**: Every sync/deploy script follows deploy-musician.sh 4-stage pattern. Learn one, know all.
  - **JSON manifests at intersections**: app-meta.json, state.json — machine-readable state.
  - **Zero proprietary infrastructure**: No servers, no databases, no cloud functions.
  </AccordionItem>

  <AccordionItem title="3. Exit Strategy">
  Survival technique is not just about running the factory. It is about being able to leave:

  - **IP transfer**: Character brands (channeling output) can be sold independently
  - **Account migration**: Fork repos to new account, update tokens, factory runs
  - **Revenue separation**: Each YouTube channel + commerce storefront operates independently
  - **Knowledge base**: 28 repos = structured RAG dataset, sellable as educational content

  The goal is not permanent operation. The goal is building a system that survives your departure.
  </AccordionItem>

  <AccordionItem title="4. Variable OS — The Invitation">
  PhoneParis (F-Droid repository) carries the tagline: "You are a variable. You can become one too."

  Variable OS means:
  - Parksy is a variable in the system, not a constant
  - The system works with any operator (variable replacement)
  - Each new operator brings their own perspective (channeling) into the same infrastructure
  - 10-20 year horizon — not quick revenue, structural survival
  </AccordionItem>
</Accordion>

## The 10-20 Year Breath

Survival technique operates on a fundamentally different timescale than typical content creation. There is no launch date, no growth hack, no viral moment to chase.

The metric is structural: can each node (person) in the system achieve life success over 10-20 years? The factory is the infrastructure that enables that timeline — not by generating quick revenue, but by providing a complete content manufacturing system that can be operated, transferred, or exited at any point.

The process of building toward this — including the parts that are not yet proven, the franchise model that has not yet been externally validated — is itself the curriculum. We broadcast the attempt, not the achievement. Public failure is how judgment forms.
