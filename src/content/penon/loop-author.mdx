## In Simple Terms

Imagine a person who builds a car factory.

But here is the twist — the factory tour IS the product. People come to watch the cars being made, and THAT is the business.

The blueprints become the assembly line. The assembly line becomes the showroom. The showroom becomes the commercial. The commercial brings customers. The customers fund new blueprints.

A Loop Author is exactly this. They do not separate "writing" from "coding" from "broadcasting" from "business." Each output feeds directly into the next stage, and the last stage feeds back into the first.

There is no final product. The loop IS the product.

Traditional creators have a pipeline: idea → create → ship → done.
A Loop Author has a circuit: idea → write → code → app → broadcast → revenue → idea.

Nothing is wasted. Nothing is separate. Everything is everything else.

## System Architecture

<Accordion>
  <AccordionItem title="1. Why existing labels fail">

| Label | Scope | Where it ends |

Each label describes a segment. None describes the loop.

A Loop Author operates across ALL segments simultaneously, and more importantly, the output of each segment is the input of the next.
  </AccordionItem>

  <AccordionItem title="2. The Loop Structure">

The defining structure of a Loop Author:

**Stage 1: Writing** — A whitepaper, a manifesto, a design document. This is not "content" yet. It is raw thought crystallized into text.

**Stage 2: Code** — The writing becomes architecture. The document becomes a system. The idea becomes executable.

**Stage 3: App** — The code becomes a product. Users can touch it, use it, experience it.

**Stage 4: Broadcast** — The process of building becomes content. The factory tour. The making-of. The development log itself is the show.

**Stage 5: Business** — The broadcast generates audience. The audience generates revenue. The revenue funds the next loop.

**Stage 6: Writing again** — Revenue and audience feedback become the raw material for the next whitepaper. The loop closes. The loop restarts.

The critical insight: these stages are not sequential phases of a project. They run simultaneously and continuously.
  </AccordionItem>

  <AccordionItem title="3. Infrastructure = Content">

For a Loop Author, infrastructure decisions are content decisions.

- Choosing Git over Google Docs is not a technical preference — it is a narrative structure (commits are sentences, branches are chapters).
- Building a deployment pipeline is not DevOps — it is creating the printing press.
- Writing a CLAUDE.md is not documentation — it is a constitution for the publishing house.

The stack IS the story. The toolchain IS the medium.
  </AccordionItem>

  <AccordionItem title="4. The Franchise Template">

A Loop Author does not just run one loop. They design loops that can be replicated.

One repository = one factory. But the factory blueprint itself can be stamped onto new repositories, new domains, new verticals.

28 repositories are not 28 separate projects. They are 28 branches of the same franchise, each running its own loop, connected by cross-repo manifests and synchronization scripts.
  </AccordionItem>

## Theoretical Background

The Loop Author model emerges from a simple observation: in the digital age, the boundaries between creation, distribution, and monetization have collapsed.

A traditional author writes a book. A publisher prints it. A distributor sells it. A marketer promotes it. These are separate roles, separate organizations, separate skill sets.

A Loop Author compresses all of these into a single person operating a single system. But compression alone does not make it new — what makes it new is the loop. Each stage feeds the next. The act of coding IS the content. The act of broadcasting IS the marketing. The act of doing business IS the research for the next piece of writing.

This is not "wearing many hats." It is recognizing that in a sufficiently integrated system, there is only one hat. The medium is not text, not code, not video, not business. The medium is the loop itself.

### Key Principles

**1. No stage is terminal.** Every output is an input to another stage. A deployed app is not "done" — it is raw material for broadcast content.

**2. The process is the product.** The factory tour model. Building in public is not a marketing strategy — it is the core content production method.

**3. Infrastructure is authorship.** Choosing tools, designing pipelines, writing configuration files — these are creative acts with narrative consequences.

**4. Replication over scaling.** Instead of making one loop bigger, stamp the blueprint onto new domains. Each loop stays small and fast.

**5. Git is memory.** The Loop Author does not need to remember. The commit history IS the memory. The repository IS the autobiography.
