# Part 1 — Grandpa Mode

- 개발자는 코더가 아니라, AI들을 굴리는 “공장장(Manager)”가 된다.
- 기획·지시는 모바일에서, 무거운 빌드는 전기세만 드는 집 PC가 대신 한다.
- 결과물은 두 가지: 웹에서 바로 쓰는 PWA, 단말기에 꽂히는 APK.
- GitHub는 중앙 허브, AI 셋(ChatGPT/Gemini/Claude)은 역할 분담형 라인 인력.
- 클라우드는 “처음부터”가 아니라, “커진 다음”에만 쓴다.
- 목표는 구독료·클라우드 비용을 0에 가깝게 줄인 1인 소프트웨어 제조 공장이다.

# Part 2 — System Architect Mode

## 1. Problem Framing: 왜 Patchtech인가

모바일·클라우드·마켓 각각의 구조적 한계를 전제로 출발한다.

- **모바일의 한계**  
  - 입력(필기, 터치, S Pen)에는 최적인데, 빌드·컴파일에는 OS가 막혀 있다.
  - “만들기 좋은 기기”가 아니라 “쓰게 만드는 기기”에 가깝다.

- **클라우드의 한계**  
  - Cloud IDE는 멋있지만, 결국 **시간 × 사용량 = 영구 구독료** 구조다.
  - 실험이 늘어날수록, “고정비 + 변동비”가 같이 비대해진다.

- **앱 마켓의 한계**  
  - Play Store / App Store 심사는 **대중용 서비스** 기준에 맞춰져 있다.
  - 혼자 쓰는 워크플로우 툴, 실험용 도구를 계속 올리기에는 과하다고 느껴진다.

Patchtech은 이 세 가지를 뒤집어서, **“내가 쓰기 위한 공장”**을 기준으로 설계한다.

---

## 2. Patchtech R&R Stack

Patchtech은 “코드를 잘 짠다”가 아니라,  
**AI가 짜준 조각(Patch)을 상황에 맞게 선정·조립하는 아키텍처**다.

### 2.1 Role & Responsibility Table

| Layer       | Component   | Device                | Function                                               |
|------------|------------|-----------------------|--------------------------------------------------------|
| Command    | User (CEO) | Mobile (Galaxy Tab)   | 아이디어 발상, 우선순위 결정, 결과물 검수 및 실제 사용 |
| Strategy   | ChatGPT    | Cloud                 | 맥락 분석, 백서·요구사항 정의서, 구조 설계 초안        |
| Architecture | Gemini   | Cloud                 | 기술 타당성 검증, API/스택 선택, 작업지시서 최적화     |
| Development | Claude MCP | Desktop (Local)      | 코드 생성·수정, 디렉토리 작업, 리팩토링, 자동화 스크립트 |
| Build & CI | Local PC   | Desktop (Local)       | PWA 빌드, APK 컴파일, 패키징, 아카이빙                 |
| Repository | GitHub     | Cloud                 | 버전 관리, 협업·배포 허브, 장기 백업                   |

핵심 포인트는:

- **모바일 = 지시만 하는 리모컨**
- **로컬 PC = “전기세만 드는 빌드 서버”**
- **클라우드 = GitHub + 필요할 때만 켜는 확장 슬롯**

---

## 3. Dual Track Output: PWA & APK

Patchtech은 결과물을 항상 **두 트랙**으로 생각한다.

### 3.1 Track A — PWA (Progressive Web App)

**목적**

- 데이터 시각화, 텍스트 도구, 학습 대시보드, 간단 유틸리티.
- “설치 없이 URL만 던져서 쓰는 툴”에 최적화.

**스택 & 배포**

- **Stack**: HTML/JS, 또는 Flutter Web.
- **Infra**: GitHub → Vercel 연동, 무료 호스팅.
- **업데이트**: main 브랜치 푸시 = 실시간 배포.

**장점**

- 브라우저만 있으면 OS 상관없이 접속 가능.
- GitHub Pages / Vercel로 **Zero 클라우드 요금** 운영.
- 나중에 도메인·SEO·수익화(광고, 멤버십)까지 이어지기 좋다.

---

### 3.2 Track B — Native APK (Android Package)

**목적**

- S Pen, 카메라, 센서, 로컬 파일 같은 **단말 특화 기능**이 필요할 때.
- FFmpeg 기반 영상 편집, 오디오 믹싱 등 고부하 작업.

**스택 & 배포**

- **Stack**: Flutter (공용 코드베이스) + Android 빌드.
- **Build Node**: 집/사무실의 로컬 PC + Claude MCP.
- **배포**:
  - USB / Wi-Fi Sideload (본인·소수 사용자용)
  - Google Drive / Dropbox 링크 공유 (실험용 배포)

**장점**

- 앱스토어 심사를 안 거치고도, **자기 생태계 안에서만 돌리는 개인 툴 생산** 가능.
- 완전 오프라인 구동, 로컬 파일에 바로 접근.

---

## 4. Zero-Cost Build System: 돈 안 쓰는 DevOps

Patchtech의 인프라 설계 핵심은 **“빌드만큼은 로컬에서”**이다.

### 4.1 Claude MCP Desktop = 로컬 공장장

- Claude MCP는 **PC 파일시스템에 직접 접근**해서:
  - 레포 생성 / 폴더 구조 세팅
  - 코드 수정 / 리팩토링
  - 빌드 스크립트, FFmpeg 래퍼, PWA 세팅 등 자동화
- 사용자는 모바일에서 **작업지시서(Instruction)**만 던진다.

### 4.2 Build Node로서의 Local PC

- 집에 두는 PC는 사실상 **“항상 대기 중인 무료 빌드 서버”**다.
- 해야 할 일:
  - `npm run build`, `flutter build apk`, `ffmpeg` 배치 작업,
  - GitHub push, 릴리즈 아카이브 생성.

**비용 구조**

- 클라우드 빌드 요금: **0원**
- 추가 구독료: **0원**
- 남는 건 전기세 + 인터넷 회선.

---

## 5. Cloud Scalability: 커지면 그대로 올린다

Patchtech은 **“처음부터 클라우드를 쓰지 않는다”**가 아니라,  
**“커지기 전까지는 쓸 이유가 없다”**에 가깝다.

- 로컬에서 잘 돌아가고,
- GitHub 레포가 이미 깨끗하게 관리되고 있다면,

언젠가:

- GitHub Actions
- AWS CodeBuild
- Cloud Run / Functions

같은 곳으로 **지금 구조를 그대로 올리면 된다.**

즉,

> “아이디어 검증과 개인 생산성 단계 = 로컬 + Vercel로 충분,  
> 트래픽·사용자가 붙는 순간부터 클라우드 확장.”

이 구분을 시스템 레벨에서 명시하는 게 Patchtech의 설계 철학이다.

---

## 6. Value Proposition: 무엇이 자산이 되는가

Patchtech이 만드는 가치는 코드 조각 자체가 아니다.

1. **Software Sovereignty (소프트웨어 주권)**  
   - 남이 만든 앱의 기능 제한에 갇히지 않는다.
   - “내가 필요한 기능만 있는 도구”를 내가 설계해서 쓴다.

2. **Assetization (워크플로우의 자산화)**  
   - 완성된 PWA / APK + 그걸 만드는 **Instruction 세트**가 하나의 번들 자산.
   - 강의, 수강생 패키지, 팬덤 굿즈처럼 묶어서 제공 가능.

3. **Speed (병목 제거)**  
   - “코딩을 공부해야만 만든다”는 병목을 제거.
   - 말 → 구조화(백서) → AI 코드 → 로컬 빌드 → 설치까지 **짧은 루프**로 회전.
