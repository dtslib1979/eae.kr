# Part 2 – 핵심 내용

## 핵심 결론 (요약)

| 한 줄 요약 | 의미 |
|---|---|
| 상한선은 삼성 안드로이드 | 루팅 없이 가능한 자동화·개발의 최대치 |
| UI는 못 고친다 | One UI / Good Lock은 옵션만 쓴다 |
| 보안은 건드리지 않는다 | 루팅 안 하면 Knox·금융앱 안전 |
| 백업은 재현이다 | 상태 저장 ❌ / 명령 재실행 ⭕ |

---

## 우리가 처음에 착각했던 것들

### 1️⃣ “Good Lock 좀 더 파면 UI 고칠 수 있지 않을까?”
→ ❌ 불가  

Good Lock은 삼성이 허용한 옵션 UI  
코드·로직·흐름 수정 불가  
바꿀 수 있는 건 “값”이지 “구조”가 아님

---

### 2️⃣ “Shizuku면 시스템도 만질 수 있지 않나?”
→ ❌ 과대평가  

Shizuku = ADB 권한 중계기  
설정 값 접근은 가능  
UI 조작, 전역 훅, 시스템 개조는 불가  
재부팅 시 권한 해제됨

---

### 3️⃣ “업데이트만 막으면 내가 왕 아닌가?”
→ ❌ 근본 해결 아님  

업데이트 차단은 임시  
언젠가 강제됨  
버전 고정은 리스크 누적

---

### 4️⃣ “스냅샷 백업이면 끝 아닌가?”
→ ❌ 여기서 가장 많이 망함  

OS·보안 영역 포함 안 됨  
업데이트 후 복원 충돌  
기기 바뀌면 의미 없음

---

## 플랫폼 철학 차이 (여기서 게임이 갈림)

### PC (Windows / macOS)

관리자 권한 = 거의 신  
고장 나면 사용자 책임

### Samsung Android

수억 대 단위 기기  
보안 사고 = 회사 존망  
그래서 시스템은 철저히 잠금  

👉 대신:  
> 사용자 영역은 마음껏 써라

---

## 도구별 “현실적인” 위치

### Termux
- 사용자 공간 리눅스  
- 개발·스크립트·자동화 가능  
- 초기화 대비 → 스크립트 재실행

### Shizuku
- ADB 권한 앱 중계  
- 설정 값 접근  
- UI 조작 ❌  
- 재부팅 시 재페어링 필요

### Tasker (+ 접근성)
- UI 자동화 담당  
- 클릭·흐름 흉내 가능  
- 로직 백업 가능  
- 권한은 매번 재허용

### Claude Code
- 이 판의 핵심  
- 상태 저장이 아니라  
  → “지금 환경 기준으로 다시 만드는 법”에 특화

---

## 절대 불가 리스트

루팅 없이는 아무리 삽질해도 안 되는 것들:

- One UI / 설정 앱 코드 수정  
- Good Lock 로직 변경  
- 시스템 앱 교체  
- 커널·드라이버 접근  
- /system /vendor 쓰기  
- 전역 제스처 재정의  
- Knox·보안 키 접근  
- 무제한 백그라운드 데몬  
- OS 업데이트 완전 통제  
- 디스크 이미지 백업  

👉 이거 시도하면 시간 낭비다.

---

## 그럼 어디까지는 되나?

| 범주 | 가능 여부 |
|---|---|
| 시스템 값 변경 | ⭕ (노출된 것만) |
| UI 자동화 | ⭕ (접근성 기반) |
| 오버레이 UI | ⭕ (겉만) |
| 사용자 앱 자동화 | ⭕ |
| 시스템 구조 변경 | ❌ |
